## F2. 로컬 환경 이미지 불러오기 (Metashape 방식)

### 개발 의도
**이번 스프린트의 핵심 항목.** Metashape가 이미지를 불러오는 방식과 동일하게, 사용자가 폴더 경로를 지정하면 시스템이 해당 경로의 이미지를 직접 읽어오도록 한다.

현재는 로컬 환경에서도 브라우저가 파일을 읽어 HTTP로 전송하는 불필요한 과정을 거친다. Metashape는 `chunk.addPhotos(파일경로_목록)`으로 디스크에서 바로 읽는데, 우리 시스템만 HTTP를 경유하여 같은 파일을 디스크에 2번 기록하고 있다.

### 현재 상태 (문제점)
```
현재 흐름:
  브라우저 → HTTP (10MB 청크) → 스토리지 디스크 저장 (2회 쓰기)
  → 처리 시작 시 _prepare_images()가 스토리지에서 symlink 생성
  → Metashape가 symlink 경로로 이미지 읽음

실제 Metashape 처리 방식:
  chunk.addPhotos(["/path/img1.jpg", "/path/img2.jpg"])  ← 경로만 있으면 됨
```
- Metashape 엔진은 이미지 **파일 경로 목록**만 받으면 동작 (align_photos.py:46)
- `_prepare_images()`(tasks.py:177)가 스토리지 파일을 처리 디렉토리에 symlink 생성
- 로컬 환경에서는 원본 이미지가 이미 디스크에 있으므로, HTTP 업로드 → 스토리지 저장이 불필요

### 목표 흐름
```
개선 흐름 (Metashape 방식):
  사용자가 폴더 경로 입력 (예: /mnt/external-hdd/block-A/)
  → 백엔드가 경로를 스캔하여 이미지 목록 확인
  → Image 레코드 생성 (original_path = 원본 위치)
  → 처리 시작 시 _prepare_images()가 원본 경로에서 바로 symlink 생성
  → Metashape가 읽음

결과: 파일 복사/이동 없음, 경로 등록만 수행 → 즉시 완료
```

### 개발 단계

**BE-1. 이미지 경로 등록 API**
- `POST /api/v1/upload/projects/{project_id}/local-import`
- 요청 본문: `{ source_dir: "/mnt/external-hdd/block-A/images" }`
- 사용 시나리오:
  - 외장하드를 서버에 꽂고 마운트된 경로 지정 (예: `/mnt/external-hdd/...`)
  - 로컬 마운트된 하드디스크의 경로 지정 (예: `/data/raw/block-A/...`)
- 동작:
  1. source_dir 스캔 → 이미지 파일 목록 확인 (확장자 필터: jpg/jpeg/tif/tiff/png)
  2. 각 파일에 대해 `Image` 레코드 생성
     - `original_path` = 원본 파일의 절대 경로 (파일 이동 없음)
     - `upload_status` = `"completed"`
     - `file_size` = 원본 파일 크기
  3. 기존 `_prepare_images()`는 `original_path`에서 symlink를 생성하므로 수정 불필요
- **파일 복사/이동 없음** — 경로만 DB에 등록

**BE-2. 경로 유효성 검증**
- source_dir 존재 여부 확인
- 이미지 파일이 1개 이상 존재하는지 확인
- path traversal 방지 (`.../../../etc` 등)
- 허용 경로 화이트리스트: 환경변수 `LOCAL_IMPORT_ALLOWED_PATHS`로 제한
- 단일 사용자 로컬 환경이므로 최소한의 보안

**BE-3. 처리 시 원본 경로 접근 가능 여부 확인**
- `_prepare_images()` 실행 시 `original_path` 파일이 여전히 존재하는지 검증
- 외장하드가 분리되었거나 파일이 삭제된 경우 명확한 에러 메시지
- 기존 로직: `storage.get_local_path()` → `os.symlink()` — 이미 경로 기반이므로 호환

**FE-1. 기존 업로드 UI 유지, 내부 동작만 변경**
- 기존 업로드 화면 레이아웃 유지 ("폴더 선택" / "이미지 선택" 카드 구조 그대로)
- "폴더 선택" 클릭 시: 경로 입력 필드 표시 → 사용자가 폴더 경로 입력 → `/local-import` API 호출
- "이미지 선택" 클릭 시: 경로 입력 또는 파일명 목록 입력 → 동일하게 경로 등록
- 내부 동작만 변경: 기존 HTTP 청크 전송 → 경로 문자열만 백엔드에 전달
- 응답: 등록된 이미지 수, 총 크기 표시

### 성능 기대치
| 시나리오 | 현재 (HTTP 업로드) | 개선 후 (경로 등록) | 비고 |
|----------|-------------------|-------------------|------|
| 1000장(50GB) | ~25분 | **< 3초** | 파일 복사 없음, DB 레코드 생성만 |
| 외장하드 3000장(150GB) | ~75분 | **< 5초** | 동일 |

### 검증 방법
- [ ] 로컬 하드디스크 경로 입력 → 이미지 목록 즉시 등록 확인 (< 5초)
- [ ] 외장하드 경로 입력 → 동일하게 등록 확인
- [ ] 등록 후 처리 시작 → `_prepare_images()`에서 원본 경로 symlink 정상 생성 확인
- [ ] Metashape가 symlink를 통해 이미지를 정상 읽는지 확인
- [ ] 등록 후 외장하드 분리 → 처리 시작 시 명확한 에러 메시지 확인
- [ ] 존재하지 않는 경로 입력 → 에러 메시지 확인
- [ ] path traversal 경로 → 차단 확인
- [ ] 가져오기 완료 후 Image 레코드 정상 생성 + 썸네일 생성 트리거 확인

### 개발 리포트 기록 항목
- 기존 HTTP 업로드 vs 경로 등록 방식 성능 비교
- `original_path` 필드 활용: 기존 `_prepare_images()` 호환성 확인 결과
- 경로 보안 검증 정책 및 화이트리스트 설정 방법
- 외장하드 분리 시나리오 대응 (처리 시점에 원본 접근 불가 시 처리)
- 기존 업로드 UI 재사용 범위: 레이아웃 유지 + 내부 동작 변경 지점 목록
- EO 업로드는 기존 수동 방식 유지 (경로 등록 후 별도 EO 업로드 필요 → UI에 안내 표시)

## F3. 프로젝트 업로드 시 즉시 처리 예약

### 개발 의도
프로젝트를 생성하고 이미지를 업로드하는 시점에, **"업로드 완료되면 바로 처리"** 옵션을 걸어놓을 수 있게 한다. 사용자가 프로젝트 하나를 만들고 업로드+처리예약을 걸고, 바로 다음 프로젝트를 만들고 또 업로드+처리예약을 거는 식으로 **야간 무인 운영**을 가능하게 한다. 모든 예약된 처리는 동일 Celery 큐에서 순차 소화된다.

### 현재 상태
- `ProcessingJob.status == "scheduled"` 상태가 이미 존재
- upload.py(851-920행)에 scheduled → queued 전환 로직 구현됨
- 그러나 **scheduled Job을 생성하는 프론트엔드 진입점이 없음**
- 현재 워크플로우: 업로드 완료 → 수동으로 처리 시작 버튼 클릭 필수

### 사용자 시나리오 (야간 무인 운영)
```
19:00  프로젝트 A 생성 → 이미지+EO 업로드 → "완료 후 처리" 체크
       → scheduled Job 생성
19:05  프로젝트 B 생성 → 이미지+EO 업로드 → "완료 후 처리" 체크
       → scheduled Job 생성
19:10  프로젝트 C 생성 → 이미지+EO 업로드 → "완료 후 처리" 체크
       → scheduled Job 생성
19:15  퇴근
19:20  A 업로드 완료 → scheduled → queued → processing 시작
20:30  A 처리 완료 → B 업로드 이미 완료 → queued → processing 시작
21:40  B 처리 완료 → C processing 시작
22:50  C 처리 완료
06:00  출근 → 전부 완료 확인
```

### 개발 단계

**BE-1. 처리 예약 API (단일 프로젝트)**
- `POST /api/v1/processing/projects/{project_id}/schedule`
- 기존 `/start` API와 유사하나 **업로드 완료 여부를 검사하지 않음**
- `ProcessingJob(status="scheduled")` 생성
- 업로드 완료 + EO 존재 상태라면 즉시 `queued`로 설정
- 이미 scheduled/queued/processing Job이 있으면 409 Conflict

**BE-2. 업로드 완료 시 scheduled → queued 전환**
- 기존 upload.py의 전환 로직 활용 (수정 최소화)
- 전환 조건: 프로젝트 내 모든 이미지 `completed` + EO 데이터 존재
- EO 미존재 시 scheduled 유지 + `project.pending_reason = "eo_required"` 설정
- 로컬 가져오기(F2) 완료 시에도 동일한 전환 로직 트리거

**BE-3. EO 미존재 시 처리 차단 (기존 `/start` + 신규 `/schedule` 공통)**
- 기존 `/start` API에도 EO 존재 여부 검사를 추가 (현재는 검사 없음)
- `/schedule` API에도 동일 검사 적용
- **EO 미업로드 상태에서 처리/예약 요청 시**: 거부 + 경고 반환
  - HTTP 422 + `"EO 파일이 업로드되지 않았습니다. EO 업로드 후 처리를 시작해주세요."`
- 이유: EO 없이 Metashape 처리를 시작하면 정렬 실패 또는 무의미한 결과 생성

**FE-1. 업로드 화면에 "완료 후 처리" 체크박스**
- 프로젝트 업로드 화면(UploadWizard)에 체크박스 추가
- 기본값: ON (로컬 단일 사용자 환경 기본 동작)
- 체크 시 이미지 업로드와 동시에 `/schedule` API 호출
- **EO 미업로드 시**: 체크박스 비활성화 + 경고 메시지 "EO 파일을 먼저 업로드해주세요"
- EO 업로드 완료 후 체크박스 자동 활성화

**FE-2. 예약 상태 표시**
- 프로젝트 카드/목록에 `scheduled` 상태 배지 표시
- scheduled 상태에서 미충족 조건 표시 (예: "업로드 중 3/10")
- 처리 시작 시 토스트 알림

### 검증 방법
- [ ] 이미지+EO 업로드 + "완료 후 처리" 체크 → 업로드 완료 시 자동 처리 시작 확인
- [ ] "완료 후 처리" 미체크 → 업로드 완료 후 수동 시작 필요 확인
- [ ] **EO 미업로드 상태에서 "완료 후 처리" 체크 시도 → 체크 불가 + 경고 메시지 확인**
- [ ] **EO 미업로드 상태에서 `/schedule` API 호출 → 422 에러 확인**
- [ ] **EO 미업로드 상태에서 기존 `/start` API 호출 → 422 에러 확인 (기존 동작 변경)**
- [ ] EO 업로드 완료 후 체크박스 활성화 → 체크 → 예약 정상 동작 확인
- [ ] 3개 프로젝트를 순차적으로 생성+예약 → 모두 동일 큐에서 순차 처리 확인
- [ ] 이미 처리 중인 프로젝트에 재예약 시도 → 409 에러 확인
- [ ] F2(로컬 가져오기) 완료 시에도 scheduled → queued 전환 정상 동작 확인

### 개발 리포트 기록 항목
- `/schedule` API와 기존 `/start` API의 차이점 및 공존 방식
- EO 미존재 시 처리 차단: `/start`와 `/schedule` 모두에 적용된 검사 로직 기록
- 기존 `/start` API 동작 변경 영향: EO 없이 처리 시작하던 기존 사용 패턴이 있는지 확인
- scheduled → queued 전환 조건 (이미지 완료 확인)
- F2(로컬 가져오기) 완료 이벤트가 전환 트리거로 정상 연결되는지
- 야간 무인 운영 E2E 테스트 시나리오 (3개 프로젝트 순차 생성+예약 → 전체 완료)

## F4. 베이스맵 품질관리원 보유 데이터 교체

### 개발 의도
현재 오프라인 모드에서 OpenStreetMap 기반 타일을 사용한다. 품질관리원이 보유한 XYZ 타일 데이터로 교체하여 실무 환경에 맞는 베이스맵을 제공한다.

### 현재 상태
- `mapConfig.js`에서 `VITE_MAP_OFFLINE=true` 시 `/tiles/{z}/{x}/{y}` 경로 사용
- nginx가 로컬 타일 디렉토리를 서빙 (XYZ 구조)
- 타일 URL 템플릿, attribution, maxZoom은 환경변수로 설정 가능
- 기존 오프라인 타일 교체 문서 존재 (커밋 `17f61ef`)
- **품관원 데이터는 XYZ 타일 포맷** → 기존과 동일 구조이므로 포맷 변환 불필요

### 개발 단계

**OPS-1. 타일 디렉토리 교체**
- 품관원 XYZ 타일을 nginx 서빙 경로에 배치 (기존 타일 디렉토리 교체 또는 병행)
- 줌 레벨 범위 확인 및 `mapConfig.js`의 minZoom/maxZoom 조정
- 타일 확장자(png/jpg) 확인 → nginx 자동 감지 설정과 일치하는지 점검

**OPS-2. (선택) 다중 타일 레이어 경로 분리**
- 복수 타일셋(항공사진, 지형도 등) 보유 시: `/tiles/aerial/`, `/tiles/topo/` 등 경로 분리
- F7(베이스맵 on/off)과 연계하여 레이어 선택 가능하도록 확장

### 검증 방법
- [ ] 오프라인 환경에서 품관원 타일이 정상 렌더링되는지 확인
- [ ] 줌 레벨 범위 전체에서 타일 누락 없는지 확인
- [ ] 기존 프로젝트 footprint/정사영상 오버레이와 타일 좌표계 정합 확인

### 개발 리포트 기록 항목
- 타일 용량 및 줌 레벨 범위 기록
- 교체 전후 지도 스크린샷 비교
- F7(베이스맵 on/off) 연계: 다중 레이어 제공 여부 결정

## F5. 도엽 단위 클립+머지 기능

### 개발 의도
정사영상을 국가 표준 도엽(지도 격자) 단위로 잘라내거나(클립), 여러 도엽을 하나로 합치는(머지) 기능을 제공한다. 이는 납품/검수 시 도엽 단위 산출물이 요구되기 때문이다.

### 현재 상태
- GDAL 유틸리티 존재: `gdalinfo`, `extract_bounds_wkt`, `gdalwarp` (재투영/리샘플링)
- 클립/머지 기능 없음
- 다운로드 시 `gdalwarp`으로 좌표계/해상도 변환은 가능

### 개발 단계

**BE-1. 도엽 격자 데이터 관리**
- 표준 도엽 격자 데이터(1:1000, 1:5000 등) DB 테이블 또는 GeoJSON 파일로 관리
- 도엽 ID, 좌표 범위(bounds), 축척 정보 저장
- API: `GET /api/v1/sheets?scale=1000&bounds=...` — 영역 내 도엽 목록 조회

**BE-2. 클립 기능 (단일 정사영상 → 도엽 단위 분할)**
- `gdalwarp -cutline {도엽_polygon} -crop_to_cutline` 또는 `gdal_translate -projwin` 사용
- 입력: 프로젝트 COG + 도엽 ID 목록
- 출력: 도엽별 GeoTIFF 파일
- Celery 태스크로 비동기 처리 (대용량 정사영상)

**BE-3. 머지 기능 (다중 정사영상 → 단일 결과)**
- `gdal_merge.py` 또는 `gdalwarp` 활용한 모자이킹
- 입력: 여러 프로젝트의 정사영상 + 대상 영역(도엽 또는 사용자 정의 범위)
- 출력: 병합된 GeoTIFF
- 겹침 영역 처리 정책: 최신 프로젝트 우선 또는 사용자 선택

**BE-4. 클립+머지 결과 관리**
- 결과물을 임시 디렉토리에 저장, 다운로드 토큰 발급
- 결과 만료 정책: 24시간 후 자동 삭제
- 진행률 WebSocket 전송 (대용량 처리 시)

**FE-1. 도엽 격자 오버레이**
- 처리 완료 프로젝트의 지도 위에 도엽 격자 표시
- 축척 선택 드롭다운 (1:1000, 1:5000 등)
- 정사영상과 겹치는 도엽 자동 하이라이트

**FE-2. 클립/머지 작업 UI**
- 도엽 선택 (클릭 또는 영역 드래그)
- "도엽별 내보내기" 버튼 → 클립 API 호출
- "선택 도엽 병합" 버튼 → 머지 API 호출
- 작업 진행 상태 표시 + 완료 시 다운로드 링크

### 검증 방법
- [ ] 도엽 격자 오버레이가 정사영상 위에 정확히 정렬되는지 확인
- [ ] 단일 정사영상을 3개 도엽으로 클립 → 각 결과의 범위가 도엽 경계와 일치하는지 확인
- [ ] 2개 프로젝트 정사영상을 1개 도엽 범위로 머지 → 결과물에 두 영상이 포함되는지 확인
- [ ] 클립/머지 결과의 좌표계, GSD가 원본과 일치하는지 gdalinfo로 확인
- [ ] 대용량(10GB+) 정사영상 클립 시 메모리/디스크 사용량 모니터링

### 개발 리포트 기록 항목
- 도엽 격자 데이터 출처 및 축척별 격자 수
- GDAL 명령어 벤치마크 (클립/머지 소요 시간, 메모리)
- 겹침 영역 처리 정책 결정 사항
- 결과물 임시 저장 경로 및 자동 삭제 정책
- 좌표계/GSD 정합성 검증 결과

## F6. 내보내기 후 썸네일 생성 및 COG 삭제 선택

### 개발 의도
정사영상 내보내기(다운로드) 완료 후, 대시보드 표시용 썸네일을 생성하고 디스크 절약을 위해 원본 COG 삭제 여부를 사용자가 선택할 수 있게 한다. COG 삭제 시 TiTiler 타일 서빙은 불가해지지만, 썸네일로 위치/범위를 확인할 수 있다.

### 현재 상태
- 썸네일 워커: 원본 이미지(항공사진)용 256×256 JPEG 생성
- 정사영상용 썸네일 생성 기능 없음
- COG 삭제 로직 없음 (MinIO 업로드 후 로컬 COG만 삭제)
- 다운로드 API에서 후처리 로직 없음

### 개발 단계

**BE-1. 정사영상 썸네일 생성**
- COG에서 overview 레벨을 읽어 축소 이미지 생성 (GDAL `gdal_translate -outsize`)
- 출력: 프로젝트별 `ortho_thumbnail.jpg` (512×512 이하, JPEG 85%)
- bounds 정보를 썸네일 메타데이터에 포함 (JSON sidecar)
- 저장 경로: `projects/{project_id}/thumbnails/ortho_thumb.jpg`
- Project 모델에 `ortho_thumbnail_path` 필드 추가

**BE-2. COG 삭제 API**
- `DELETE /api/v1/projects/{project_id}/ortho/cog`
- 삭제 전 조건: 정사영상 썸네일이 이미 존재해야 함
- 삭제 시: COG 파일 삭제 + `Project.ortho_path = null` 설정
- 삭제 후에도 유지되는 정보: bounds, GSD, 면적, 썸네일
- 되돌릴 수 없음을 응답에 명시

**BE-3. 내보내기 완료 후 삭제 제안 흐름**
- 다운로드 완료 이벤트 감지: 다운로드 스트리밍 정상 종료 시 플래그 설정
- `Project.last_exported_at` 타임스탬프 기록
- 프론트엔드에서 내보내기 완료 후 "COG 삭제" 제안 다이얼로그 표시

**FE-1. 내보내기 완료 후 다이얼로그**
- 다운로드 완료 시 "정사영상 원본(COG)을 삭제하시겠습니까?" 확인 다이얼로그
- 삭제 시 절약되는 디스크 용량 표시
- "삭제" / "유지" 선택
- 삭제 시 재확인 ("삭제하면 지도에서 타일 보기가 불가합니다")

**FE-2. 대시보드 썸네일 표시**
- COG 존재 시: 기존 TiTiler 타일 표시
- COG 삭제 후: 썸네일 이미지를 bounds에 맞춰 ImageOverlay로 표시
- 삭제 상태 표시: 프로젝트 카드에 "원본 삭제됨" 배지

### 검증 방법
- [ ] 처리 완료 프로젝트의 정사영상 썸네일이 정상 생성되는지 확인
- [ ] 썸네일이 원본 정사영상의 영역/방향과 일치하는지 확인
- [ ] 내보내기 완료 후 삭제 다이얼로그 표시 → "삭제" 선택 → COG 파일 삭제 확인
- [ ] COG 삭제 후 대시보드에서 썸네일 오버레이 정상 표시 확인
- [ ] COG 삭제 후 TiTiler 타일 요청 시 적절한 에러/폴백 처리 확인
- [ ] 썸네일 미존재 상태에서 COG 삭제 시도 → 거부 확인

### 개발 리포트 기록 항목
- 정사영상 썸네일 생성 방식 (overview 활용 vs 전체 리샘플링) 성능 비교
- COG 삭제 전후 디스크 사용량 변화 측정
- TiTiler 폴백 처리 방식 (404 반환 vs 썸네일 리다이렉트)
- ImageOverlay 성능 (대시보드에 다수 프로젝트 썸네일 표시 시)
- F5(클립+머지)와의 연계: COG 삭제된 프로젝트의 클립/머지 가능 여부 정책

## F7. 대시보드/처리화면 베이스맵 on/off 토글

### 개발 의도
대시보드 지도와 처리화면 지도 모두에서 베이스맵 레이어를 표시하거나 숨길 수 있게 한다. 정사영상 위에 베이스맵이 불필요하거나 방해되는 경우 끌 수 있다.

### 현재 상태
- `FootprintMap.jsx`(대시보드)에서 TileLayer를 고정 렌더링
- `ProjectMap.jsx`(처리화면)에서도 TileLayer 고정 렌더링
- 레이어 컨트롤 없음
- Leaflet의 `LayersControl` 미사용

### 개발 단계

**FE-1. 베이스맵 토글 공통 컴포넌트**
- 재사용 가능한 `BasemapToggle` 컴포넌트 생성
- 지도 우상단에 레이어 토글 버튼 배치
- `TileLayer`의 `opacity`를 0/1로 전환하거나, 조건부 렌더링
- 상태: localStorage에 저장하여 세션 간 유지 (키: `basemap_visible`)

**FE-2. 대시보드 지도(FootprintMap)에 적용**
- `FootprintMap.jsx`에 `BasemapToggle` 컴포넌트 삽입

**FE-3. 처리화면 지도(ProjectMap)에 적용**
- `ProjectMap.jsx`에 `BasemapToggle` 컴포넌트 삽입
- 처리화면에서는 정사영상/EO 마커와 함께 동작해야 하므로 레이어 순서 확인

### 검증 방법
- [ ] **대시보드**: 토글 off → 베이스맵 숨겨지고 정사영상만 표시 확인
- [ ] **대시보드**: 토글 on → 베이스맵 복원 확인
- [ ] **처리화면**: 토글 off → 베이스맵 숨겨지고 정사영상/EO 마커만 표시 확인
- [ ] **처리화면**: 토글 on → 베이스맵 복원, 정사영상/EO 마커와 레이어 순서 정확 확인
- [ ] 페이지 새로고침 후 토글 상태 유지 확인 (대시보드/처리화면 공통)
- [ ] 두 화면 간 토글 상태 동기화 확인 (대시보드에서 off → 처리화면에서도 off)

### 개발 리포트 기록 항목
- 토글 구현 방식 (opacity vs 조건부 렌더링) 선택 근거
- 공통 컴포넌트 분리 방식 및 적용 대상 목록
- localStorage 키 네이밍 컨벤션
- 처리화면에서의 레이어 z-index 순서 (베이스맵 < 정사영상 < EO 마커)

## F8. 처리화면 지도 범위 확장 및 복귀 버튼

### 개발 의도
처리화면(ProjectMap)에서 지도가 정사영상 bounds에 고정되어 주변 지역을 볼 수 없다. 사용자가 자유롭게 지도를 이동/축소하여 바운더리 외 영역을 확인하고, 버튼 클릭으로 원래 정사영상 범위로 돌아올 수 있게 한다.

### 현재 상태
- `ProjectMap.jsx`의 `FitBounds` 컴포넌트가 EO 좌표 또는 project.bounds로 자동 fit
- `maxBounds` 설정은 없으나, `FitBounds`가 지도 이동/줌 변경 시마다 재실행될 가능성
- 지도 자유 이동이 제한되는 원인 파악 필요

### 개발 단계

**FE-1. FitBounds 동작 수정**
- 초기 로드 시에만 bounds에 fit (이후 사용자 조작 허용)
- `FitBounds` 컴포넌트에 `once` 플래그 추가 또는 `useEffect` 의존성 조정
- `maxBounds` 속성이 있다면 제거

**FE-2. "범위 복귀" 버튼 추가**
- 지도 위 컨트롤 영역에 "범위 복귀" 버튼 배치 (아이콘: 🎯 또는 홈 아이콘)
- 클릭 시 `map.fitBounds(projectBounds, { padding: [50, 50] })` 호출
- 현재 뷰가 이미 bounds 내에 있으면 버튼 비활성화 (선택)

### 검증 방법
- [ ] 처리화면 진입 → 정사영상 범위에 자동 fit 확인
- [ ] 이후 지도 드래그/줌아웃으로 범위 밖 이동 가능 확인
- [ ] "범위 복귀" 버튼 클릭 → 원래 정사영상 범위로 복귀 확인
- [ ] 정사영상 없는 프로젝트(EO만 있는 경우) → EO 범위로 복귀 확인

### 개발 리포트 기록 항목
- FitBounds 기존 동작 분석 결과 (재실행 트리거 조건)
- 버튼 위치/스타일 스크린샷
- 다른 지도 컴포넌트(FootprintMap 등)에 동일 패턴 적용 필요 여부
